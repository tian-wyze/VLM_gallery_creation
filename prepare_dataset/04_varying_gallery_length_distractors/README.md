# 04 — Varying Gallery Length with Hard-Negative Distractors

Prepare train/test splits for VLM-based person re-identification (ReID) where the gallery size varies and is padded with **hard negatives** mined from DINOv2 embeddings.

## Task

Given a **query image** (captured by a home security camera) and a **gallery** (one image per household member plus hard-negative distractors from other households), the model must decide which gallery person matches the query — or output `-1` to indicate the query is a **stranger** (distractor) who matches no one.

## Input

Two household-info JSON files generated by `clean_gallery.py` (or the cleaning pipeline in `03_clean_gallery_images/`):

```
household_info_v2_same_clothes.json
household_info_v2_cross_clothes.json
```

Each has the structure:

```json
{
  "<household_id>": {
    "<identity_id>": {
      "<mac_addr>": {
        "query": ["<image_path>", ...],
        "gallery": ["<image_path>", ...]
      }
    }
  }
}
```

## Usage

```bash
cd IDA-VLM/prepare_dataset/04_varying_gallery_length_distractors

# Default run (seed=42, max gallery=5, uses cached embeddings if present)
python3 varying_gallery.py

# Override max gallery size or custom household files
python3 varying_gallery.py --max_gallery 7
python3 varying_gallery.py --same_clothes /path/to/same.json --cross_clothes /path/to/cross.json
```

On first run, DINOv2 ViT-L/14 embeddings are computed for every image (requires GPU) and cached to `cached_embeddings/dinov2_embeddings.pt`. Subsequent runs reuse the cache.

## Case Format

Every case (test and training) has the same structure:

```json
{
  "query": "/absolute/path/to/query.jpg",
  "gallery": [
    "/absolute/path/to/img1.jpg",
    "/absolute/path/to/img2.jpg",
    "/absolute/path/to/img3.jpg"
  ],
  "label": 2,
  "similarity": [0.31, 0.82, 0.47],
  "query_household_id": "10026312",
  "household_id": ["255785_4528992", "10026312", "820315"]
}
```

| Field | Meaning |
|---|---|
| `query` | Query image path |
| `gallery` | List of gallery image paths (shuffled) |
| `label` | `> 0`: 1-indexed position of the match. `-1`: stranger, no match |
| `similarity` | DINOv2 cosine similarity between query and each gallery image (same order) |
| `query_household_id` | Household of the query person |
| `household_id` | List parallel to `gallery`; household origin of each image — lets you identify which gallery entries are hard negatives from other households |

## Hard-Negative Mining

Hard negatives are padded into galleries using DINOv2 ViT-L/14 embeddings. The `HardNegativeMiner` precomputes top-200 nearest-neighbor indices for every image in a pool (one pool per split) via batched matrix multiply, then each per-case lookup is O(1) + a small filter pass.

**Selection rules** for each hard negative:

1. Drawn from other households (train negatives come from train households only, test from test — no leakage).
2. One image per identity in the final list (diversity — improvement #4).
3. Query image and excluded households never appear in the result.
4. **Stochastic sampling**: the miner collects the top-K (default K=10) most-similar valid candidates, then randomly samples N from them. This keeps negatives hard while adding variety across cases with the same query.

## Gallery Size (variable)

For each case, a random target size is drawn in `[natural_size, max_gallery]`, then the gallery is padded with hard negatives:

| Case type | Natural size | Target size distribution |
|---|---|---|
| Non-distractor singleton | 1 | uniform in [1, 5] |
| Non-distractor family-of-N | N | uniform in [N, 5] (or exactly N if N ≥ 5) |
| Distractor singleton | — | uniform in [1, 5], filled with top-K similar views of the resident |
| Distractor family-of-N | N | uniform in [N, 5], padded with hard negatives from other households |

This matches the goal: train the model to handle any gallery size from the natural household size up to `max_gallery`.

## Test Benchmarks (10 subsets, 1,600 cases total)

### 8 non-distractor subsets

Each varies along three dimensions:

| Dimension | Values | Meaning |
|---|---|---|
| **Clothes** | `sameclothes`, `crossclothes` | Query and gallery from same or different clothing events |
| **Household** | `singleton`, `family` | 1 person vs. 2+ persons |
| **Camera** | `samecamera`, `crosscamera` | All images from same MAC or different MAC |

Each subset targets 50 unique query identities and 150 cases. Each case:
- **query**: 1 image of the query person
- **gallery**: 1 image per household member + hard negatives up to random `target_size`
- **label**: 1-indexed position of the matching person in the shuffled gallery

### 2 distractor subsets

- `distractor_cropped_singleton` — stranger query + 1-to-5 images of the resident (selected as the top-K most similar to the stranger)
- `distractor_cropped_family` — stranger query + family members + hard negatives from other households

Both use `label = -1`. 200 cases each, stratified by family size.

## Training Data

Training cases are built from all training households across both clothes types. Three training-only improvements are applied:

| # | Improvement | Description |
|---|---|---|
| **1** | **Role swapping** | Query and gallery image pools are merged per (identity, MAC), so any image can serve as query or gallery input. Effectively doubles the usable image pool. |
| **3** | **Hard-negative padding** | Galleries are padded with DINOv2-mined hard negatives (stochastically sampled from top-10 similar candidates in other households). Replaces random padding. |
| **6** | **Gallery diversity tracking** | A `UsageTracker` prefers unused images per identity, falling back to reuse only when all images have been seen. |

### Sampling rates

| Family size | Cases per identity | Rationale |
|---|---|---|
| 1 (singleton) | 30 | Abundant identities; moderate resampling |
| 2 | 40 | Most common family size |
| 3 | 30 | Capped to avoid over-representation |
| 4+ | 30 | Rare; capped at 30 so no single identity dominates |

**Distractor training cases**: 20% of total, stratified by family size.

## Train/Test Split

Households are classified into three tiers using info from **both** clothes types:

| Tier | Definition | Why it matters |
|---|---|---|
| Family shared-MAC | Family where ≥2 members share a camera MAC | Required for `family_samecamera` tests |
| Family cross-only | Family with no shared MAC | Can only contribute to `family_crosscamera` tests |
| Singleton | 1 identity | Gallery size is always 1 |

The split is performed once at the household level and applied to both clothes types — no household appears in both train and test. Tier lists are **sorted before shuffling** (improvement #8) so the split is fully deterministic given the seed, regardless of Python's internal hash randomization. Split metadata is saved to `train_test_split.json`.

### Split distribution

| Tier | Test | Train | Total |
|---|---|---|---|
| Family shared-MAC | 30 | 42 | 72 |
| Family cross-only | 15 | 101 | 116 |
| Singleton | 70 | 158 | 228 |
| **Total** | **115** | **301** | **416** |

### Data leakage verification

By construction:

| Level | Train | Test | Overlap |
|---|---|---|---|
| Households | 301 | 115 | 0 |
| Identities | split at household level | split at household level | 0 |
| Images | hard-negative pool = train-only | hard-negative pool = test-only | 0 |

## Known Data Limitations

- **`family_samecamera` subsets have few identities**: same-camera requires *all* household members to have gallery images on the query's MAC, which is strict and reduces the number of valid cases.
- **Gallery sizes > 5 are rare in test**: only families with 5–6 members exceed the cap; those naturally keep their full size with no padding.
- **`singleton_crosscamera` in same\_clothes is thin**: only ~75 singletons in same\_clothes have images from >1 camera.
